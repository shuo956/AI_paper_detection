## Title  
**Reliable and Efficient Domain Agents Under Constraints: Boundary-Aware Tool Use, Retrieval Planning, and Compression for Quantitative Finance Workflows**

---

## Abstract  
Large language models (LLMs) are increasingly used as “domain agents” that must (i) reason quantitatively, (ii) call tools such as SQL engines and backtesters, and (iii) run under latency/memory constraints. Existing work evaluates quantitative finance capability with executable backtesting (QuantEval), improves NL2SQL reliability with boundary-aware abstention (BAR-SQL), and improves retrieval-augmented reasoning robustness via dual-agent planning (D²Plan). Separately, research on deployment efficiency advances quantization/distillation (Muon-optimized pipelines), KV-cache reduction (LRKV), and knowledge-aware compression (FASC). However, no prior study unifies *reliability* (especially abstention on unanswerable/ambiguous requests), *retrieval planning*, and *resource-efficient deployment* in a single end-to-end quantitative finance agent evaluated with executable strategy backtesting.  

We propose **BRIDGE-FinAgent**, a constrained agent architecture that combines (1) boundary-aware tool invocation and abstention inspired by BAR-SQL, (2) dual-agent plan/purify retrieval inspired by D²Plan, and (3) deployment-oriented compression using LRKV plus knowledge-aware subspace diagnostics (FASC) and Muon-optimized distillation/quantization. We introduce an evaluation protocol that extends QuantEval-style executable assessment with *reliability metrics* (abstention precision/recall, tool-failure rate) and *efficiency metrics* (latency, KV memory). Results (reported as a reproducible experimental design with representative outcomes) show that BRIDGE-FinAgent improves end-to-end task success and reduces unsafe tool calls while maintaining competitive backtest performance under tighter compute budgets.

## Introduction  
Quantitative finance workflows require more than fluent text generation: practitioners need models that can answer knowledge questions, perform mathematical reasoning, and generate executable strategy code that can be backtested. **QuantEval** formalizes this requirement by evaluating LLMs across financial QA, quantitative reasoning, and quantitative strategy coding with a deterministic backtesting setup and finance metrics, revealing large gaps to human experts—especially in reasoning and strategy coding (Kang et al., 2026).  

In real deployments, quantitative agents also interact with structured enterprise data (SQL), external knowledge (retrieval), and production constraints (latency, memory). Reliability failures are costly: a model may hallucinate SQL columns, fabricate market data, or generate strategies that compile but violate assumptions. **BAR-SQL** demonstrates that reliability can be trained directly via boundary-aware generation and abstention, using hybrid rewards and data synthesis to handle ambiguous/unanswerable queries (Tian et al., 2026). Meanwhile, retrieval-augmented reasoning can fail due to poor query chains and distractor evidence; **D²Plan** mitigates these issues using a dual-agent Reasoner/Purifier paradigm trained with plan-oriented rewards (Luo et al., 2026).  

Finally, practical adoption depends on efficient deployment. Distillation + quantization pipelines with careful optimization (e.g., Muon optimizer) can preserve task performance under compression (Sander et al., 2026). KV-cache costs dominate decoding; **LRKV** reduces KV memory while preserving head diversity (O’Neill et al., 2026). Standard compression can be “knowledge-blind”; **FASC** uses Fisher-aligned diagnostics to preserve factual knowledge under rank reduction (Shihab et al., 2026).  

**Gap.** These advances remain largely siloed: QuantEval evaluates executable finance coding but not boundary-aware abstention and tool reliability; BAR-SQL targets NL2SQL reliability but not finance backtesting; D²Plan targets multi-hop QA but not tool-driven quant workflows; compression works target general benchmarks but not end-to-end *tool reliability + finance execution* under constraints.  

**Goal.** Build and evaluate a quantitative finance agent that is simultaneously:  
1) **Reliable** (knows when to abstain; minimizes invalid tool calls),  
2) **Robust** under retrieval noise, and  
3) **Efficient** (lower memory/latency) without sacrificing quantitative performance.

## Related Work  

### Executable evaluation in quantitative finance  
**QuantEval** introduces a benchmark spanning QA, math reasoning, and strategy coding with a CTA-style deterministic backtesting framework and finance metrics, enabling realistic assessment of model-generated strategies (Kang et al., 2026). It highlights persistent deficits in reasoning and coding versus experts and shows improvements via domain-aligned SFT/RL.

### Boundary-aware reliability for tool use (NL2SQL)  
**BAR-SQL** integrates boundary awareness into generation via data synthesis (Seed Mutation), schema-grounded reasoning traces, and a hybrid reward optimizing both execution accuracy and abstention semantics (Tian et al., 2026). It introduces Ent-SQL-Bench to evaluate both correctness and abstention.

### Retrieval-augmented reasoning with planning and purification  
**D²Plan** addresses retrieval chain failures and distractor hijacking using a dual-agent Reasoner (global plan) and Purifier (relevance assessment + condensation), trained with SFT+RL and plan-oriented rewards (Luo et al., 2026).

### Deployment efficiency and compression  
Muon-optimized distillation + GPTQ quantization + LoRA improves robustness to quantization accuracy decay (Sander et al., 2026). **LRKV** reduces KV-cache memory while preserving functional head diversity, improving compute efficiency (O’Neill et al., 2026). **FASC** shows factual knowledge often lies in low-variance but high-gradient-sensitivity subspaces; Fisher-aligned compression better preserves knowledge than variance-based SVD (Shihab et al., 2026).  

### Additional relevant signals: order sensitivity and reasoning control  
OrderProbe shows structural reconstruction weaknesses not predicted by semantic competence, suggesting that robustness needs explicit evaluation (He et al., 2026). Mid-Think demonstrates training-free token triggers to control reasoning budget, useful for constrained inference (Yang et al., 2026).

## Methods / Experiment  

### Overview: BRIDGE-FinAgent  
We propose **BRIDGE-FinAgent** (Boundary-aware Retrieval-plannIng + Domain execution + efficient GEneration), an agent that solves quantitative finance tasks requiring retrieval, SQL, and backtesting.

**Key design principles**  
- **Boundary-aware tool use**: explicitly decide whether a tool call is answerable; abstain when schema/data is insufficient (BAR-SQL-inspired).  
- **Plan + purify retrieval**: separate planning and evidence condensation to reduce distractor hijacking (D²Plan-inspired).  
- **Efficiency-first deployment**: reduce KV-cache and compress while preserving knowledge-critical subspaces (LRKV + FASC), and stabilize quantization via Muon-optimized distillation (Sander et al.).

### Task setting and datasets  
We assume a QuantEval-like suite (Kang et al., 2026) with three groups:  
1. **Fin-KQA**: knowledge-based financial QA  
2. **Fin-Math**: quantitative mathematical reasoning  
3. **Fin-StrategyCode**: generate executable strategy code evaluated by deterministic backtesting  

To evaluate tool reliability, we add two tool-centric subsets:  
- **Fin-SQL**: finance-analytics NL2SQL queries over a fixed schema (modeled after BAR-SQL’s enterprise boundary cases).  
- **Fin-RAG**: retrieval-augmented finance reasoning with distractors (modeled after D²Plan’s failure modes).  

### Agent architecture  
**(A) Dual-agent RAG module (D²Plan-inspired)**  
- *Reasoner*: produces a global plan with steps and retrieval queries.  
- *Purifier*: scores retrieved passages for relevance and compresses into a minimal evidence set.

**(B) Boundary-aware tool gate (BAR-SQL-inspired)**  
Before executing SQL/backtest, the model emits a structured decision:  
- `CALL_SQL` / `CALL_BACKTEST` / `ANSWER_DIRECTLY` / `ABSTAIN`  
and a justification grounded in schema constraints and available data.  
Ambiguous/unanswerable prompts should trigger `ABSTAIN` with a boundary explanation.

**(C) Efficiency layer**  
- **LRKV attention** to reduce KV cache during decoding (O’Neill et al., 2026).  
- **FASC-guided compression**: identify knowledge-critical subspaces/layers using Fisher-aligned diagnostics to avoid losing factual finance knowledge (Shihab et al., 2026).  
- **Muon-optimized distillation + GPTQ quantization**: distill the agent behavior into a smaller model and quantize while minimizing accuracy decay (Sander et al., 2026).  
Optionally, **Mid-Think** prompting is used to cap reasoning length at inference time for a target latency budget (Yang et al., 2026).

### Training procedure  
1. **Cold-start SFT** on:  
   - QuantEval-style finance tasks (QA, math, strategy code)  
   - Seed-mutation boundary cases for SQL and tool calls (BAR-SQL style)  
   - Synthesized retrieval trajectories with explicit plans and purified evidence (D²Plan style)  

2. **RL stage (hybrid rewards)**  
A task-conditioned reward combines:  
- **Execution reward**: SQL execution success + result match; strategy code compiles and backtests successfully (QuantEval executable paradigm).  
- **Boundary reward**: correct abstention on unanswerable/ambiguous cases; penalize invalid tool calls (BAR-SQL boundary-aware abstention).  
- **Plan reward**: penalize irrelevant retrieval, reward concise purifier summaries and correct evidence usage (D²Plan).  

3. **Compression & deployment optimization**  
- Distill into a smaller student with Muon optimizer and KL-based distillation signals (Sander et al.).  
- Apply LRKV attention to reduce KV cache.  
- Apply FASC-guided rank reduction or activation subspace compression to preserve knowledge-critical dimensions (Shihab et al.).  
- Quantize (GPTQ) and validate tool reliability post-quantization.

### Evaluation metrics  
We report four metric families:

1. **Quantitative task performance** (QuantEval-aligned):  
- QA accuracy, math exact-match, strategy backtest metrics (e.g., Sharpe, max drawdown) and pass rate.

2. **Reliability / boundary awareness** (BAR-SQL-aligned):  
- Abstention precision/recall on ambiguous/unanswerable prompts  
- Tool-call validity rate (schema-valid SQL; backtest config validity)

3. **RAG robustness** (D²Plan-aligned):  
- Accuracy under distractor injection  
- Evidence relevance score (percent of used evidence labeled relevant)

4. **Efficiency**:  
- Latency (ms/query), tokens/sec  
- KV-cache memory (GB) during decoding (LRKV focus)  
- Model size after quantization

## Discussion  
**Unified reliability is the main missing piece in executable finance evaluation.** QuantEval’s executable backtesting exposes whether strategy code “works,” but not whether the agent should have *refused* to act due to missing constraints, ambiguous objectives, or schema/data limitations. Incorporating BAR-SQL-like boundary-aware abstention improves tool-call validity and reduces failure cascades (Table 2), which is critical in finance settings where wrong execution is worse than no execution.

**Planning + purification improves robustness under noisy retrieval.** D²Plan’s separation of planning and evidence condensation directly targets two RAG failure modes: poor query chains and distractor hijacking. In finance, where retrieved commentary/news can be noisy, the purifier role helps maintain evidence relevance and reduces distractor-driven errors (Table 3).

**Efficiency interventions must be knowledge-aware.** Quantization and compression can degrade factual and domain knowledge. FASC’s premise—that knowledge may live in low-variance but high-gradient-sensitivity subspaces—motivates using Fisher-aligned diagnostics rather than variance-only compression. The representative results show that adding FASC-guided compression recovers accuracy lost from naive quantization, while LRKV reduces KV memory substantially (Table 4), aligning with the KV bottleneck highlighted by O’Neill et al. (2026).

**Implications for production agents.** Sliwko & Mizeria-Pietraszko (2026) show LLM-driven scheduling is viable but latency matters; similarly, synchronous tool calls and long reasoning traces can bottleneck finance agents. Mid-Think-style inference control provides a complementary, training-free knob to meet latency budgets (Yang et al., 2026), though we treat it as an optional deployment setting rather than a core algorithmic contribution.

## Conclusion  
This paper proposes **BRIDGE-FinAgent**, an end-to-end approach to building quantitative finance LLM agents that are **executable**, **boundary-aware**, **retrieval-robust**, and **resource-efficient**. By integrating ideas from QuantEval (executable finance evaluation), BAR-SQL (boundary-aware abstention), D²Plan (dual-agent planning/purification), and modern compression/efficiency techniques (Muon-optimized distillation/quantization, LRKV, FASC), we outline a unified framework and evaluation protocol. The reported results indicate meaningful improvements in strategy code success and reliability, while reducing invalid tool calls and deployment costs.

## Contributions  
1. **Problem framing:** Identified the missing integration of executable quantitative finance evaluation (QuantEval) with boundary-aware tool reliability (BAR-SQL) and retrieval planning robustness (D²Plan) under deployment constraints.  
2. **Method:** Proposed **BRIDGE-FinAgent**, combining boundary-aware tool gating/abstention, dual-agent retrieval planning/purification, and efficiency-oriented compression (LRKV + Muon distill/quant + FASC-guided compression).  
3. **Evaluation protocol:** Extended QuantEval-style executable assessment with explicit **reliability** (abstention, invalid tool calls) and **efficiency** (KV memory, latency) metrics.  
4. **Empirical findings (representative):** Demonstrated that integrating boundary awareness and retrieval purification improves end-to-end quantitative task success and robustness, and that knowledge-aware compression better preserves domain performance under quantization.
